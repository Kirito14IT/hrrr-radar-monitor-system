# 毫米波雷达睡眠看护系统页面与功能说明

> 本文档整理当前 Web App 各页面定位、功能说明、数据来源和演示讲解口径。内容按最终项目展示目标撰写：系统由毫米波雷达开发板与呼噜检测开发板组成，前端 Web App 负责展示、分析、预警和历史回看。

## 1. 项目整体说明

| 项目项 | 说明 |
|---|---|
| 项目名称 | 毫米波雷达睡眠看护与呼噜扰动监测系统 |
| 应用场景 | 卧室睡眠监测、养老看护、夜间值守、健康状态展示、无接触生命体征采集 |
| 核心硬件 | 毫米波雷达开发板、呼噜检测开发板 |
| 核心能力 | 非接触心率/呼吸率监测、呼噜扰动识别、睡眠稳定性分析、离床/无人判断、设备在线状态监测、AI/本地健康报告 |
| Web App 角色 | 作为最终展示端，提供项目首页、实时监测、睡眠驾驶舱、看护预警中心、历史数据与 AI 洞察 |
| 数据融合方式 | 将雷达数据和音频/呼噜特征按时间戳对齐，统一进入后端时间轴，再由前端分页面展示 |

## 2. 页面总览表

| 页面顺序 | 页面名称 | 路由 | 页面定位 | 面向对象 | 核心讲解重点 |
|---:|---|---|---|---|---|
| 1 | 项目首页 | `/manage/project_intro` | Web App 首页与项目简介 | 评委、老师、演示观众、项目接手者 | 说明系统最终形态、双开发板传感器、数据流、页面模块 |
| 2 | 生命体征监测 | `/manage/heart_pic` | 实时监测主页面 | 使用者、调试者、演示者 | 展示雷达生命体征、呼噜板状态、实时趋势和健康摘要 |
| 3 | 睡眠看护驾驶舱 | `/manage/sleep_dashboard` | 睡眠分析大屏 | 看护人员、演示观众 | 用评分、热力图、事件流解释整段睡眠质量 |
| 4 | 看护预警中心 | `/manage/alert_center` | 夜间异常响应页面 | 看护人员、值守人员 | 将异常变成可执行的看护动作队列 |
| 5 | 历史数据 | `/manage/data` | 历史记录与 AI 分析 | 使用者、研究人员、答辩展示 | 查询历史生命体征并生成本地/AI 分析报告 |

## 3. 页面详细功能讲解

### 3.1 项目首页

| 讲解维度 | 内容 |
|---|---|
| 页面目标 | 作为整个 Web App 的入口，向第一次打开系统的人解释“这是什么项目、用了哪些硬件、能解决什么问题”。 |
| 核心信息 | 项目面向睡眠看护；使用毫米波雷达开发板和呼噜检测开发板；最终形成非接触监测、睡眠分析、预警响应和历史报告闭环。 |
| 主要模块 | 顶部项目介绍、双开发板数据流示意、核心价值卡片、最终项目工作流程、开发板照片占位符、Web App 页面入口。 |
| 硬件展示 | 预留两张照片位置：毫米波雷达开发板实拍图、呼噜检测开发板实拍图。 |
| 交互功能 | 可点击进入实时监测、睡眠驾驶舱、预警中心和历史数据页面。 |
| 适合讲解的话术 | “这个首页不是调试页，而是最终项目展示页，用来说明双传感器如何采集数据、后端如何融合、前端如何形成看护决策。” |

### 3.2 生命体征监测

| 讲解维度 | 内容 |
|---|---|
| 页面目标 | 实时展示当前人的心率、呼吸率、距离、呼噜强度和设备状态，是系统最直观的实时监测页面。 |
| 数据来源 | 毫米波雷达开发板提供心率、呼吸率、目标距离、相位波形；呼噜检测开发板提供呼噜分数、音频片段、呼噜事件。 |
| 核心组件 | 开始/停止实时监测按钮、雷达板在线状态、呼噜板在线状态、心率趋势图、呼吸趋势图、呼噜强度趋势图、声浪条、睡眠分期、健康摘要。 |
| 关键能力 | 心率/呼吸/呼噜使用同一时间轴展示；雷达离线或无人时图表断线，不用 0 伪装真实数据；呼噜板离线不会影响雷达生命体征显示。 |
| 适合场景 | 实时演示开发板数据变化、验证双板是否在线、展示心率和呼吸率随时间变化。 |
| 适合讲解的话术 | “这一页展示的是实时数据层，能看到雷达板和呼噜板是否在线，以及两路传感器在同一个时间轴上的同步效果。” |

### 3.3 睡眠看护驾驶舱

| 讲解维度 | 内容 |
|---|---|
| 页面目标 | 把实时数据进一步整理成睡眠质量、扰动趋势和夜间事件，适合展示系统的分析能力。 |
| 数据来源 | `/sleep/overview` 接口，融合时间轴数据、设备状态、呼噜事件、睡眠事件和历史统计。 |
| 核心组件 | 睡眠质量评分圆环、评分扣分原因、雷达/呼噜板状态胶囊、呼噜扰动地图、夜间守护事件流、稳定性卡片。 |
| 关键能力 | 将心率稳定性、呼吸稳定性、呼噜强度、离床/无人、设备异常综合成 0-100 分睡眠质量评分。 |
| 交互功能 | 实时看护/历史回放切换、时间范围选择、事件过滤、刷新。 |
| 适合场景 | 展示整晚睡眠质量、说明呼噜对生命体征的扰动、解释系统如何记录夜间事件。 |
| 适合讲解的话术 | “这一页不是只看原始曲线，而是把曲线转成睡眠评分、扰动热力和事件流，更接近实际看护使用。” |

### 3.4 看护预警中心

| 讲解维度 | 内容 |
|---|---|
| 页面目标 | 面向看护人员，将复杂数据转化为风险等级和具体动作建议。 |
| 数据来源 | `/sleep/overview`、`/timeline`、`/status`，分别提供分析概览、实时趋势和设备状态。 |
| 核心组件 | 风险总览卡、雷达/呼噜板状态卡、异常归因矩阵、照护动作队列、阈值策略面板、趋势小窗。 |
| 异常类型 | 心率异常、呼吸异常、呼噜扰动、离床/无人、设备链路异常。 |
| 关键能力 | 根据阈值策略判断风险，生成“检查雷达距离”“确认呼噜板在线”“观察呼吸波动”等可执行动作。 |
| 持久化功能 | 阈值策略和已处理动作保存在前端本地状态，刷新页面后仍保留。 |
| 适合场景 | 夜间值守、异常响应演示、说明系统如何从监测走向看护决策。 |
| 适合讲解的话术 | “预警中心把系统输出从‘数据显示’提升为‘下一步该做什么’，更接近真实看护产品。” |

### 3.5 历史数据与 AI 洞察

| 讲解维度 | 内容 |
|---|---|
| 页面目标 | 查询历史生命体征记录，并基于当前页数据生成 AI 或本地规则健康报告。 |
| 数据来源 | `/heartdata/selectPage` 查询历史心率/呼吸率；`/ai/analyze-vitals` 生成 DeepSeek 或本地规则报告。 |
| 核心组件 | 日期筛选、查询按钮、AI 分析按钮、重置按钮、数据质量 KPI、历史表格、AI/本地健康分析卡片。 |
| 表格能力 | 保留数据编号、用户编号、日期、心率、呼吸率字段，并对心率/呼吸率增加“偏低/正常/偏高”标签。 |
| AI 状态修复 | AI 分析状态已移入 Pinia Store，用户点击 AI 分析后跳转到其他页面，再返回历史数据页，分析中状态或最终报告不会丢失。 |
| 适合场景 | 展示历史数据闭环、说明系统有长期记录能力、生成答辩或演示用健康摘要。 |
| 适合讲解的话术 | “这一页解决的是历史回看和报告生成；AI 请求不再绑死在页面组件里，所以路由切换不会丢失分析过程。” |

### 3.6 呼吸监测

| 讲解维度 | 内容 |
|---|---|
| 页面目标 | 独立展示呼吸相关波形，用于观察雷达相位或呼吸趋势。 |
| 数据来源 | 后端实时接口中的呼吸率和相位数据。 |
| 核心组件 | 呼吸波形图、呼吸频率显示。 |
| 适合场景 | 算法调试、硬件联调、单独验证呼吸提取效果。 |
| 适合讲解的话术 | “这是偏调试和算法观察的页面，可以辅助说明雷达如何从微动信号中提取呼吸信息。” |

## 4. 数据流与功能链路

| 阶段 | 输入 | 处理 | 输出 | 对应页面 |
|---|---|---|---|---|
| 硬件采集 | 雷达回波、音频信号 | 两块开发板分别提取生命体征与呼噜特征 | 心率、呼吸率、距离、呼噜分数、音频片段 | 项目首页、生命体征监测 |
| 后端接收 | 雷达帧、呼噜 heartbeat、音频片段 | 写入共享时间轴，计算在线状态和睡眠状态 | `/timeline`、`/status`、`/sleep/overview` | 生命体征监测、驾驶舱、预警中心 |
| 实时展示 | 秒级时间轴 | 同步渲染心率、呼吸、呼噜趋势 | 实时曲线、声浪、睡眠分期 | 生命体征监测 |
| 睡眠分析 | 时间窗内的生命体征、呼噜、设备状态 | 评分、聚合、事件归因 | 睡眠质量评分、扰动地图、事件流 | 睡眠看护驾驶舱 |
| 预警决策 | 最新数据、阈值策略、设备状态 | 风险分级、动作生成 | 异常归因矩阵、照护动作队列 | 看护预警中心 |
| 历史回看 | 数据库历史记录 | 分页查询、异常标签、AI/本地规则分析 | 历史表格、健康报告 | 历史数据 |

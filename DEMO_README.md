# 睡眠看护与雷达监测系统——比赛测试与演示清单

本文档用于赛前联调、现场演示和故障降级。推荐演示主线为：**守护监测 → 多源数据看板 → 语音 SOS 全链路 → 告警解除 → 小智对话**。

> 默认后端电脑地址：`192.168.0.102`；后端 HTTP：`8081`；雷达 UDP：`9988`；前端开发端口：`5173`。

## 1. 演示目标

在五分钟内让评委看到以下闭环：

- 雷达板无接触采集人体微动，并向后端发送生命体征数据。
- Edgi Talk 默认运行守护模式，完成本地呼噜检测、求助关键词检测、温湿度上报和 IMU 跌落告警。
- FastAPI 后端融合雷达、呼噜、环境和紧急事件，并统一输出状态与时间线。
- Web 驾驶舱展示实时数据和风险事件，Android App 接收监护通知。
- Edgi Talk 可切换到对话模式，与小智大模型进行多轮语音交流。

系统用于看护辅助展示，不作为医疗诊断或真实急救系统。

## 5. 每次演示前启动流程

### 5.1 启动后端

在项目根目录运行：

```powershell
conda activate radar
python backend\realtime_radar_processing.py --port 9988 --decomp-type cwt --api-port 8081
```

等待终端出现 API 已启动信息。模型文件缺失时会影响雷达算法结果，不能只看 HTTP 服务已启动。

### 5.2 检查后端

新开 PowerShell：

```powershell
Invoke-RestMethod http://192.168.0.102:8081/status
Invoke-RestMethod "http://192.168.0.102:8081/sleep/overview?mode=live&seconds=1800"
```

- [ ] `/status` 中 `running` 为 `true`。
- [ ] Edgi Talk 上电后，`snore_board_online` 或 `edgi_board_online` 变为 `true`。
- [ ] 温湿度上报后，`environment_board_online` 为 `true`。
- [ ] 雷达开始发包后，`radar_board_online` 和 `radar_online` 状态符合现场情况。
- [ ] `active_emergency` 为空，避免带着上次告警开始演示。

### 5.3 启动前端

```powershell
cd frontend
npm run dev
```

浏览器打开 `http://localhost:5173`：

- [ ] 登录或注册成功，无 CORS 报错。
- [ ] 打开睡眠驾驶舱、预警中心、环境分析三个页面并保留标签页。
- [ ] 页面刷新后仍能恢复后端状态。

### 5.4 启动硬件与手机

1. 上电雷达板，确认串口显示 Wi-Fi 和 UDP 发送状态。
2. 上电 Edgi Talk，确认主页显示“守护模式”和关键词在线状态。
3. 打开 Android App，后端地址填入 `http://192.168.0.102:8081`，点击“开始监听”。
4. 若展示蓝牙近场数据，再扫描并连接 `RadarCare-E84`。

## 6. T-10 分钟快速通关清单

以下任一关键项失败，都应立即采用第 9 节的降级方案，不要到台上临时调代码。

### 6.1 后端与页面

- [ ] `http://192.168.0.102:8081/status` 返回 `200`。
- [ ] 登录、驾驶舱、预警中心和环境分析页可打开。
- [ ] 页面能看到设备在线状态和温湿度。
- [ ] 后端终端无端口占用、持续异常或数据库错误。

### 6.2 Edgi Talk 双模式

- [ ] 上电默认进入守护模式，呼噜检测已启动。
- [ ] 普通说话不会播放大模型回答。
- [ ] 说“救命”能出现本地 SOS 页面和报警声。
- [ ] 前端出现置顶紧急事件，Android 在 5 秒内收到通知。
- [ ] 解除后，开发板、后端、网页和 App 状态同步恢复。
- [ ] 点击“对话模式”后立即进入聆听，小智能正常回答一个短问题。
- [ ] 返回“守护模式”后对话停止，守护检测恢复。

### 6.3 雷达与融合

- [ ] 雷达 UDP 帧计数持续增加。
- [ ] 人在雷达检测区域时，目标存在/距离状态合理。
- [ ] 心率和呼吸率有值时趋势图持续更新，不出现明显跳变。
- [ ] 已验证“低呼吸 + 呼噜”联合证据可以产生疑似呼吸暂停事件。
- [ ] 单独呼噜或单个雷达异常不会误报疑似呼吸暂停。

## 7. 五分钟现场演示脚本

| 时间 | 操作 | 主讲内容 | 成功标志 |
|---|---|---|---|
| `0:00–0:25` | 展示系统全景和标题页 | 说明独居老人/睡眠看护场景，以及传统单一传感器误报高、缺少闭环的问题 | 评委理解问题与目标 |
| `0:25–0:55` | 指向雷达板、Edgi Talk、后端、Web、手机 | 用一句话解释“端侧感知—边缘融合—多端告警”架构 | 五个节点和两条链路讲清楚 |
| `0:55–1:35` | 展示 Edgi Talk 守护模式与驾驶舱 | 说明本地呼噜、云端求助词、IMU、温湿度和无接触雷达并行工作 | 页面显示设备在线和实时环境/雷达数据 |
| `1:35–2:15` | 展示驾驶舱趋势和事件流 | 说明后端不是简单转发，而是按时间线对齐多源数据并进行联合判断 | 心率、呼吸率、呼噜、环境状态可见 |
| `2:15–3:20` | 面向开发板清楚说“救命” | 展示本地 SOS → 后端事件 → Web 红色告警 → Android 通知的完整链路 | 四端在数秒内同步告警 |
| `3:20–3:45` | 在 App 或网页点击“已处理/解除” | 强调告警可追踪、可处理、可同步闭环 | `active_emergency` 消失，各端恢复 |
| `3:45–4:25` | 点击“对话模式”，问“小智，请给我一句睡前健康建议” | 说明看护终端还能按需切换为大模型陪伴入口；对话时严格暂停守护音频任务，避免资源冲突 | 小智完成 STT、LLM、TTS 回答 |
| `4:25–5:00` | 切回守护模式，展示总结页 | 总结多模态融合、端边云协同、完整告警闭环和真实硬件落地 | 回到守护主页并结束 |

演示时每个动作只讲一个结论。页面加载或语音识别等待超过 5 秒，主讲人继续介绍架构，操作员直接切换到备用方案。

## 8. 可选增强演示：疑似呼吸暂停

该演示依赖实时窗口和阈值，建议只在主流程稳定且时间充足时进行。

- [ ] 先说明策略：只有“雷达持续低呼吸”与“同时间窗呼噜证据”同时出现才告警。
- [ ] 展示单独呼噜不触发呼吸暂停，说明系统抑制误报。
- [ ] 展示联合证据事件，观察 Web 预警中心和 Android 通知。
- [ ] 强调该结果是风险提示，不是医疗诊断。

现场不等待完整窗口时，可使用提前录制的数据或演示截图，并明确说明这是同一算法链路的预采集结果。

## 9. 故障降级与应急命令

### 9.1 手动注入环境数据

```powershell
$body = @{temperature_c=24.5; humidity_pct=52.0; sensor_ok=$true; source="competition_demo"} | ConvertTo-Json
Invoke-RestMethod -Method Post -Uri http://192.168.0.102:8081/hardware/environment-heartbeat -ContentType "application/json" -Body $body
```

### 9.2 手动注入呼噜数据

```powershell
$body = @{snore_score=0.82; snore_detected=$true; dbfs=-22.0; source="competition_demo"} | ConvertTo-Json
Invoke-RestMethod -Method Post -Uri http://192.168.0.102:8081/hardware/snore-heartbeat -ContentType "application/json" -Body $body
```

### 9.3 手动触发和解除紧急事件

```powershell
$body = @{source="competition_demo"; phrase="救命"; transcript="救命，我需要帮助"} | ConvertTo-Json
Invoke-RestMethod -Method Post -Uri http://192.168.0.102:8081/emergency -ContentType "application/json" -Body $body

$resolve = @{source="competition_demo"; resolved_by="presenter"; resolution_note="现场演示解除"} | ConvertTo-Json
Invoke-RestMethod -Method Post -Uri http://192.168.0.102:8081/emergency/resolve -ContentType "application/json" -Body $resolve
```

### 9.4 常见故障处理表

| 现象 | 快速判断 | 现场处理 |
|---|---|---|
| `/status` 无法访问 | 后端未启动、端口占用或防火墙阻断 | 检查 `8081` 监听；只保留一个后端进程 |
| 浏览器显示 CORS/Network Error | 实际可能是后端 500 或旧进程 | 先直接访问 `/status`；重启后端并刷新页面 |
| 登录提示用户不存在 | MySQL 未启动后切到新的 SQLite | 现场重新注册演示账号 |
| Edgi Talk 数据不上报 | Flash 中仍保存旧 IP | 执行 `backend_cfg_set 192.168.0.102 8081` |
| 雷达板在线但无数据 | UDP 目标、端口、Wi-Fi 或模型文件异常 | 检查目标 `192.168.0.102:9988`；改用预采集结果 |
| 关键词显示离线 | 小智云或外网不可用 | 展示本地呼噜检测，并用手动 `/emergency` 演示后续闭环 |
| Android 无通知 | 通知权限、后台限制或后端地址错误 | 现场展示 App 内事件；同时保留网页告警 |
| 语音环境嘈杂 | STT 未稳定命中 | 靠近麦克风、放慢语速；第二次失败立即用手动事件 |

## 11. 评委常见问题速答

**为什么同时使用雷达和声音？**  雷达擅长无接触感知呼吸微动，声音能提供呼噜证据；两者按时间窗融合，可以降低单传感器误报。

**断网还能工作吗？**  断网时本地呼噜推理和基础界面仍可运行；云端关键词和大模型对话会离线，联网后自动恢复。

**为什么设计守护/对话两种模式？**  守护模式优先持续检测和低干扰，对话模式按需开放完整语音链路；显式切换可以避免麦克风、扬声器和模型任务争用。

**数据如何流转？**  雷达通过 UDP `9988` 发送帧，Edgi Talk 通过 HTTP `8081` 上报事件和环境数据，后端统一生成状态、趋势和告警，Web 与 Android 从同一接口读取。

**系统的创新点是什么？**  真实双板协同、多模态时间窗融合、端边云分工、守护与对话双模式，以及从检测到通知再到解除的完整闭环。

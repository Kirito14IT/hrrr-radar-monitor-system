# 睡眠看护与雷达监测系统——五分钟比赛 PPT 大纲

建议使用 **8 页、总计约 5 分钟**。PPT 只保留关键词和图，具体解释放在讲稿中；现场演示从第 5 页开始。

## 第 1 页：项目标题（0:00–0:20，20 秒）

**标题**

> 基于毫米波雷达与小智大模型的智能睡眠看护系统

**副标题**

> 多模态感知 · 端边云协同 · 紧急告警闭环

**画面建议**

- 系统全景实拍：雷达板、Edgi Talk、Web 驾驶舱和手机 App。
- 页脚放团队名称、学校和比赛名称。

**讲稿提示**

“我们的项目面向夜间睡眠和独居看护场景，通过毫米波雷达、声音、环境和 IMU 多源感知，实现从风险发现、跨端告警到人工解除的完整闭环。”

## 第 2 页：痛点与需求（0:20–0:50，30 秒）

**页面要点**

- 摄像头侵犯隐私，穿戴设备影响睡眠体验。
- 单一声音或单一呼吸阈值容易误报。
- 传统监测设备只显示数据，缺少主动求助和远程闭环。

**画面建议**

- 三个痛点图标：隐私、误报、无人响应。
- 中间突出一句话：**无接触、低打扰、能联动、可处理**。

**讲稿提示**

“我们希望系统既不拍摄用户，也不要求佩戴设备；同时不能因为一次咳嗽或一个异常采样就报警，还要确保真正的求助能及时送达监护人。”

## 第 3 页：系统架构（0:50–1:30，40 秒）

**页面要点**

```text
毫米波雷达板 ── UDP 9988 ─┐
                           ├─ FastAPI 边缘后端 ─ Web 驾驶舱
Edgi Talk ─── HTTP 8081 ──┘                  └ Android App
  ├ 本地呼噜模型
  ├ 求助关键词 / IMU / 温湿度
  └ 小智云 STT + LLM + TTS
```

**画面建议**

- 用五个大节点画一张从左到右的架构图。
- 在链路上只标 `UDP 9988`、`HTTP 8081` 和“小智云”。

**讲稿提示**

“端侧负责实时感知，边缘电脑负责多源时间对齐和事件融合，云端只承担关键词转写与对话能力。Web 和 Android 共用同一个后端状态，因此告警和解除可以同步。”

## 第 4 页：四项核心创新（1:30–2:05，35 秒）

**页面要点**

1. **无接触多模态感知**：雷达呼吸微动 + 呼噜 + 温湿度 + IMU。
2. **联合证据抑制误报**：低呼吸与呼噜在同一时间窗内共同出现才提示疑似呼吸暂停。
3. **守护/对话双模式**：持续看护与大模型陪伴互不争用音频资源。
4. **完整告警闭环**：开发板、后端、网页、手机同步触发和解除。

**画面建议**

- 四张等宽卡片，每张只放一个图标、一行标题和一行解释。

**讲稿提示**

“我们的重点不是堆叠传感器，而是让不同证据在正确的时间和模式下协同，并把检测结果真正送到监护端。”

## 第 5 页：现场演示一——持续守护（2:05–2:40，35 秒）

**现场操作**

1. 展示 Edgi Talk 主屏“守护模式 · 关键词在线”。
2. 切换到 Web 睡眠驾驶舱。
3. 指出设备在线、雷达状态、温湿度、呼噜和趋势数据。

**页面保留文字**

> 守护模式：本地呼噜检测 + 求助关键词 + 环境/IMU 上报

**讲稿提示**

“设备上电默认进入守护模式。普通交流不会触发大模型回答，断网时本地呼噜检测仍继续；联网后求助关键词监听自动恢复。”

## 第 6 页：现场演示二——SOS 全链路（2:40–3:45，65 秒）

**现场操作**

1. 面向 Edgi Talk 清楚说：“救命，我需要帮助。”
2. 展示开发板 SOS 页面和报警声。
3. 展示 Web 置顶红色告警。
4. 展示 Android 通知。
5. 在 App 或网页点击“已处理/解除”，展示各端同步恢复。

**页面保留文字**

```text
语音求助 → 本地 SOS → 后端事件 → Web 告警 → 手机通知 → 处理解除
```

**讲稿提示**

“关键词命中后，系统不只在设备端响铃，而是创建可追踪事件并通知监护人。处理动作会回写后端，所有终端同步解除，从发现到响应形成闭环。”

## 第 7 页：现场演示三——小智对话（3:45–4:25，40 秒）

**现场操作**

1. 点击 Edgi Talk 的“对话模式”。
2. 提问：“小智，请给我一句睡前健康建议。”
3. 等待语音回答后切回守护模式。

**页面保留文字**

- 守护模式：监测优先、普通语句静默。
- 对话模式：暂停守护音频任务，启用 STT、LLM 和 TTS 多轮交流。

**讲稿提示**

“同一块设备还可以成为陪伴入口。我们用显式双模式避免麦克风、扬声器和推理任务争用；结束交流后，一键回到持续守护。”

## 第 8 页：成果与展望（4:25–5:00，35 秒）

**页面要点**

- 已完成两块真实开发板、后端、Web 和 Android 的端到端联调。
- 支持呼噜、环境、语音 SOS、IMU 跌落、雷达生命体征和联合风险事件。
- 下一步：个体化基线、长期睡眠报告、更多异常动作识别和真实场景数据验证。

**结束语**

> 让监测不打扰睡眠，让每一次真正的求助都能被听见。

**讲稿提示**

“这个项目把无接触感知、边缘融合和大模型交互落到了真实硬件闭环。未来我们会继续做个体化基线和更长周期验证。谢谢各位老师。”

## PPT 制作注意事项

- 每页正文不超过 5 行，字号不小于 24 pt。
- 架构图和现场数据截图优先于长段文字。
- SOS 页用红色强调，其他页面保持蓝绿科技色，避免整套 PPT 都是警告色。
- 演示页面提前打开；PPT 中保留正常状态和告警状态截图，作为网络异常时的替代素材。
- 现场对话只问一句短问题，避免不可控的长回复。
- 演示超过预定时间时，优先保留第 6 页 SOS 闭环，压缩第 5、7 页。
